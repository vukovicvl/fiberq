# FiberQ internationalisation (i18n)

Maintainer-facing reference for FiberQ's translation pipeline: how the runtime
loader works, the rules that decide whether a string is *extractable*, the
`make` workflow, and how to add a language.

> **Contributors**, including translators, want
> [`CONTRIBUTING.md`](../CONTRIBUTING.md#translations) instead — it explains
> `.ts` files, the three contribution routes, and carries the English→French and
> English→Serbian terminology glossaries. This document is about the machinery.

> **Status:** the pipeline (loader, `Makefile` targets, `tr()` marking of
> `main_plugin.py` and `fiberq/ui/`) is in place. `fiberq/dialogs/` is **not yet
> migrated** — see [Deprecated phrase-map](#the-deprecated-phrase-map-stage-2).

## Architecture

Standard Qt i18n: strings are marked at their definition site, extracted into
per-locale `.ts` catalogues, translated, compiled to binary `.qm`, and loaded at
runtime by a `QTranslator`.

```
fiberq/i18n/
  __init__.py          runtime loader (see below)
  fiberq_<loc>.ts      translator input  — text,   COMMITTED, generated by `make i18n-update`
  fiberq_<loc>.qm      runtime catalogue — binary, COMMITTED, built by `make i18n-compile`
```

### `fiberq/i18n/__init__.py` — the loader

Deliberately self-contained: it imports **only** from the Qt bindings (plus
`qgis.core` for `QgsSettings`, guarded). It must never import from
`fiberq.utils` or any other plugin package, so it can be imported from anywhere
without a circular-import risk.

| Function | Purpose |
|---|---|
| `current_locale()` | Two-letter code from `locale/userLocale` (QGIS setting), falling back to the system locale. |
| `qm_path_for(locale, plugin_dir=None)` | Absolute path of a catalogue (may not exist). |
| `install_translator(plugin_dir)` | Loads and installs the catalogue for the current locale. Returns the `QTranslator`, or `None`. |
| `remove_translator(translator)` | Uninstalls; `None`-safe; survives a already-deleted C++ object during reload. |
| `available_languages(plugin_dir=None)` | Locale codes with a `.qm` on disk. Always includes `'en'` (source language, needs no catalogue). |
| `language_name(code)` | Display name; unknown codes are returned unchanged, so a new catalogue never crashes the language menu. |

**The caller must keep a reference to the returned translator.** Qt does not
take ownership — a garbage-collected `QTranslator` silently stops translating,
with no error. `main_plugin.py` holds it as `self.translator` (installed around
line 355, removed in `unload()` around line 2157).

Missing catalogue, failed load, and Qt refusing the install are all handled by
returning `None` and logging — the plugin falls back to English rather than
failing to start.

### Locale selection

`current_locale()` reads QGIS's own language setting first
(*Settings → Options → General → User interface translation*), not the OS
locale. This is what users expect: switching QGIS to French switches FiberQ to
French. The result is truncated to two letters, so `fr_BE` and `fr_FR` both load
`fiberq_fr.qm`. Region-specific catalogues are not currently supported.

## The `tr()` context trap — read this before touching any string

**This is the single thing that will silently break FiberQ's translations, and
it fails with no error message at any stage.** The `.ts` file looks correct, the
translator translates it, the compile succeeds, and the string still renders in
English.

Every translatable string carries a **context** (a namespace, in practice a
class name). A string is only found at runtime if the context in the catalogue
matches the context passed to `QCoreApplication.translate()` when the program
runs.

**`pylupdate6` derives the context from the enclosing class in the source file —
it does not read the context argument of a `tr()` helper you call.** So the
catalogue can be written with one context while the runtime looks up another.

### Verified behaviour

Empirically confirmed on this tree with **pylupdate6 6.10.2 / PyQt 6.10.2**.
Given this probe file:

```python
from PyQt6.QtCore import QCoreApplication


def tr(message):                                    # module-level helper
    return QCoreApplication.translate('FiberQ', message)


class Widget:
    def tr(self, message):                          # method on the class
        return QCoreApplication.translate('Widget', message)

    def build(self):
        a = self.tr("Place manhole")
        b = tr("Module level helper")
        c = QCoreApplication.translate('Explicit', "Explicit form")
        d = QCoreApplication.translate('ThreeArg', "Three arg form", None)
        return a, b, c, d


def free_function():
    return tr("Free function string")
```

`pylupdate6` extracts **four** messages:

| Source string | Context written to `.ts` | Context used at runtime | Result |
|---|---|---|---|
| `Place manhole` | `Widget` | `Widget` | ✅ translates |
| `Module level helper` | `Widget` | `FiberQ` | ❌ **silent miss** |
| `Explicit form` | `Explicit` | `Explicit` | ✅ translates |
| `Three arg form` | `ThreeArg` | `ThreeArg` | ✅ translates |
| `Free function string` | *(not extracted at all)* | `FiberQ` | ❌ **never reaches the catalogue** |

Two distinct failure modes, both silent:

1. **`tr("...")` called from a module-level helper inside a class body** is
   extracted under the *enclosing class* name, but resolves the helper's
   hardcoded context at runtime. Catalogue and runtime disagree → never
   translated.
2. **`tr("...")` called from a free function outside any class** is **dropped
   entirely** — `Free function string` does not appear in the `.ts` at all. There
   is no warning; the string simply never becomes translatable.

Failure mode 2 is the nastier one, because there is nothing in the `.ts` file to
notice is wrong.

> **Note on the three-argument form.** An earlier working assumption held that
> `pylupdate6` skips `translate(ctx, text, None)`. On 6.10.2 it does **not** — the
> three-argument call was extracted correctly. FiberQ still standardises on the
> two-argument form: the third parameter is Qt's *disambiguation* slot, not a
> place for a filler `None`, and behaviour across `pylupdate`/`lupdate` versions
> is not worth depending on.

### The two permitted patterns

**Inside a class** — give the class a `tr()` method whose hardcoded context
string is **exactly the class name**, then call `self.tr("...")`. This is the
pattern QGIS Plugin Builder generates, and it is what every FiberQ UI class does:

```python
class RoutingUI:
    def tr(self, message):
        return QCoreApplication.translate('RoutingUI', message)   # must match the class name

    def build(self):
        self.action = QAction(self.tr("Draw route"), self.iface.mainWindow())
```

If the class is renamed, **the context string must be renamed with it**, or every
one of its strings silently reverts to English.

**Outside any class** (module level, free functions) — write the explicit
two-argument call inline. No helper:

```python
QCoreApplication.translate('FiberQ', "FiberQ – Preview Map")
```

**Never create a module-level `tr()` helper.** It triggers failure mode 1 when
called from a class and failure mode 2 when called from a function. There is no
context string that makes it work.

### Current contexts in the tree

| Context | Location |
|---|---|
| `FiberQPlugin` | `fiberq/main_plugin.py` (`self.tr`, defined ~line 333) |
| `RoutingUI`, `CableLayingUI`, `ElementPlacementUI`, `DuctingUI`, `SelectionUI`, `SlackUI`, `DrawingsUI`, `ObjectsUI` | `fiberq/ui/*_ui.py` — one `tr()` per class |
| `QuickToolbar` | `fiberq/ui/quick_toolbar.py` |
| `FiberQ` | module-level strings, and the `QUICK_TOOLS` table |

Note that `fiberq/ui/base.py` is a **shared-imports module, not a base class**.
The UI group classes are plain classes with no common parent, so each one needs
its **own** `tr()` method. There is nowhere to inherit it from — and adding a
shared base class would not help anyway, since `pylupdate6` keys on the enclosing
class, not the method's owner.

## Rules for marking strings

1. **Only ever pass a string literal to `tr()`.** `self.tr(variable)` cannot be
   extracted — `pylupdate6` performs static analysis, and there is no literal to
   find. Same for f-strings and concatenation.

   ```python
   name = self.tr("Manhole")
   msg = self.tr("Placed {n} of {kind}").format(n=count, kind=name)   # correct

   msg = self.tr(f"Placed {count} manholes")                          # NOT extractable
   msg = self.tr("Placed " + str(count) + " manholes")                # NOT extractable
   ```

2. **Interpolate after translating, never before.** Use `.format()` or `%` on
   the *result* of `tr()`, so translators receive the placeholder and can move
   it — word order differs between languages.

3. **For strings defined at import time in a data table, mark with
   `QT_TRANSLATE_NOOP` and translate at use.** `QT_TRANSLATE_NOOP` marks the
   literal for extraction without translating it there and then. This matters
   for module-level tables built before the `QTranslator` is installed.
   `fiberq/ui/quick_toolbar.py` is the reference implementation:

   ```python
   # definition site, evaluated at import — marked but not translated
   QUICK_TOOLS = [
       {'id': 'pole', 'label': QT_TRANSLATE_NOOP('FiberQ', 'Place Pole'), ...},
   ]

   # use site, at initGui time — the actual lookup
   label = QCoreApplication.translate('FiberQ', tool_def['label'])
   ```

   The `translate()` call takes a variable, which is fine **only because** the
   literal is marked elsewhere with a matching context. Marking context and
   lookup context must be identical (`'FiberQ'` in both lines above).

4. **Do not translate log messages, field names, or layer names.** Stored field
   and layer names are the data schema (see [`schema.md`](schema.md)) — the
   English shown in the attribute table is a display *alias*. Translating a
   stored identifier corrupts data.

5. **Keep placeholders stable.** Renaming `{n}` to `{count}` in the source
   invalidates every existing translation of that string.

6. **Editing the English text of a marked string invalidates its translations.**
   The source text is the lookup key. After such an edit, `make i18n-update`
   shows it as a new `unfinished` entry and the old one as `vanished`. Batch
   English wording changes rather than dribbling them out.

## Workflow

Requires `pylupdate6` (Debian/Ubuntu: `pyqt6-dev-tools`, or `pip install PyQt6`).

```bash
make i18n-update    # rescan sources -> merge into fiberq/i18n/*.ts
# ... translate the .ts (Qt Linguist, or a contributor's PR) ...
make i18n-stats     # check progress
make i18n-compile   # .ts -> .qm   (needs lrelease, see below)
git add fiberq/i18n/*.ts fiberq/i18n/*.qm
```

### `make i18n-update`

Runs `pylupdate6` over every `.py` under `fiberq/` (excluding `__pycache__` and
test files) for each locale in `LOCALES` (currently `sr fr`; override with
`make i18n-update LOCALES="fr de"`).

- **Merges.** Finished translations survive; only new strings appear as
  `type="unfinished"`. Strings whose English changed or disappeared are marked
  `vanished`/`obsolete` rather than deleted.
- **Seeds new catalogues.** A missing `.ts` is first written with the
  `language` / `sourcelanguage` attributes, because `pylupdate6` does not write
  them itself — without them Qt Linguist has to ask the translator for the target
  language and plural rules are unknown. The seed is only written when the file
  is absent, so re-runs are idempotent.
- `pylupdate6` is a pure-Python rewrite that does **not** understand `.pro`
  files, which is why sources are passed on the argv.

### `make i18n-compile`

Compiles every `.ts` to `.qm`.

**`lrelease` is not installed on the current development machine** and is not
part of PyQt — it ships with the Qt tools:

```bash
sudo apt install qt6-l10n-tools       # Qt6 (QGIS 4)
sudo apt install qttools5-dev-tools   # Qt5 (QGIS 3.x)
```

The target probes `lrelease`, `lrelease-qt6`, `lrelease6`, `lrelease-qt5` in
order and **fails loudly** with those install hints if none is found — rather
than silently shipping stale or absent `.qm` files. Override explicitly with
`make i18n-compile LRELEASE=/path/to/lrelease`.

The `.qm` format is Qt-version-independent for this purpose; either `lrelease`
produces catalogues that load under both Qt5 and Qt6.

### `make i18n-stats`

Per-locale progress straight out of the `.ts` XML (pure stdlib, no extra deps):

```
locale       total    done    todo  vanished  percent
fr             412     118     294         6    28.6%
sr             412     402      10         6    97.6%
```

`total` counts live messages only — `vanished`/`obsolete` entries are reported
separately and excluded from the percentage.

### `make i18n-check`

**Advisory only; never fails.** Run automatically as a prerequisite of
`make package`. It warns when:

- a `.qm` is **not committed** — that locale will not ship;
- a `.qm` is **older than its `.ts`** — the shipped catalogue is stale.

It is deliberately `i18n-check` and *not* `i18n-compile` that gates `package`:
`package` archives `HEAD`, so running `lrelease` at that moment could not change
the zip anyway. A hard dependency would buy nothing while breaking
`make package` — and therefore `make release` — on every machine without
`qt6-l10n-tools`, including the `qgis/qgis` CI images. `i18n-check` gives the
same warning at the same moment and always exits 0.

## Why `.qm` files must be committed

`make package` builds the plugin zip with:

```
git archive --format=zip --prefix=fiberq/ -o dist/fiberq-<version>.zip HEAD:fiberq
```

Three consequences that catch people out:

1. **Only committed files ship.** A `.qm` sitting uncommitted in the working
   tree is invisible to `git archive`. The zip is built, uploads cleanly, passes
   the scan — and the plugin is untranslated for every user.
2. **Only files under `fiberq/` ship.** Anything at the repository root is
   development tooling and never reaches users. Catalogues must live in
   `fiberq/i18n/`.
3. **Both `.ts` and `.qm` are tracked.** `.ts` is the translator's source of
   truth; `.qm` is the runtime artefact. Yes, committing a build product is
   unusual — it is required here because there is no build step between `git
   archive` and the user's QGIS.

Supporting configuration, already in place:

- `.gitignore` ignores gettext artefacts (`*.mo`, `*.pot`) and carries explicit
  negations `!fiberq/i18n/*.ts` and `!fiberq/i18n/*.qm`, so a `*.ts`/`*.qm`
  pattern pasted in from another template cannot silently start ignoring them.
- `.gitattributes` marks `*.qm binary`, keeping diffs sane and preventing
  line-ending mangling.

**After translating, always:**

```bash
make i18n-compile
git add fiberq/i18n/*.ts fiberq/i18n/*.qm
```

## Adding a new language

1. Add the code to `LOCALES` in the `Makefile`:

   ```make
   LOCALES ?= sr fr de
   ```

2. Add a display name to `_LANGUAGE_NAMES` in `fiberq/i18n/__init__.py` if it
   isn't there already. Unknown codes fall back to the raw code, so this is
   cosmetic — the language menu will not break without it.

3. Generate the catalogue:

   ```bash
   make i18n-update
   ```

   This creates `fiberq/i18n/fiberq_de.ts`, seeded with `language="de"
   sourcelanguage="en"`, and fills it with every extracted string as
   `unfinished`.

4. Commit the empty `.ts` and point the translator at it. A translator can now
   use the GitHub web editor without any local setup — this is the step that
   makes route 1 in `CONTRIBUTING.md` possible, so do it *before* asking anyone
   to translate.

5. When translations come back: `make i18n-compile`, then commit `.ts` **and**
   `.qm`.

6. Consider seeding a terminology glossary for the language in
   `CONTRIBUTING.md`, as done for French and Serbian. For a domain this
   jargon-heavy it is the difference between a usable translation and a
   confusing one.

Regional variants (`fr_BE`, `pt_BR`) are **not** supported: `current_locale()`
truncates to two letters. Supporting them means changing the loader to try the
full code before the two-letter fallback.

## The deprecated phrase-map (Stage 2)

`fiberq/utils/helpers.py` (around line 464) still contains the plugin's original
ad-hoc translation scheme: `_TRANSLATION_MAP`, `_fiberq_translate`,
`translate_text`, `_apply_text_and_tooltip`, `_apply_menu_language`. It worked by
retranslating widgets in place — reading a widget's *rendered* text and looking
it up in a dict.

It is **deprecated and superseded** by this pipeline. Do not add call sites. New
user-facing strings use `tr()`.

It is retained for now because:

- `fiberq/dialogs/` still depends on it via `fiberq/utils/legacy_bridge.py`
  (which holds a **second** `_fiberq_translate` at line 47, used by
  `dialogs/bom_dialog.py` and `utils/image_watcher.py`);
- several symbols are re-exported from `fiberq/utils/__init__.py`, so removing
  them is a public API break.

Remove the whole block once `fiberq/dialogs/` is migrated (Stage 2). Note that
of the 50 entries in `_TRANSLATION_MAP`, roughly 40 are identity pairs
(English→English) — the map is doing far less than its size suggests, and the
genuine Serbian entries have been carried into the Serbian glossary in
`CONTRIBUTING.md`.

The map also has known value collisions (two source phrases are lost when the
reverse English→Serbian lookup is built, last key wins). This is preserved
as-is deliberately; it disappears with the block.

## Troubleshooting

**A string is translated in the `.ts` but shows English in QGIS.**
Almost always the context trap. Check the `<context><name>` in the `.ts` against
the context the code resolves at runtime. If the string sits in a class, the
class's `tr()` method must hardcode that exact class name. If the class was
recently renamed, that is your bug.

**A string never appears in the `.ts` at all.**
Either it is not a literal (f-string, variable, concatenation), or it is reached
through a module-level `tr()` helper from outside a class — failure mode 2 above,
which drops it silently. Mark it inline with
`QCoreApplication.translate('FiberQ', "...")`, or with `QT_TRANSLATE_NOOP` if it
is defined at import time.

**Everything is English, no matter the locale.**
Check, in order: is there a `.qm` for that locale in the *installed* plugin
directory (not just the repo)? Is it committed (`make i18n-check`)? Does QGIS's
*User interface translation* setting actually match? Is the `QTranslator`
reference still alive — a garbage-collected translator stops working silently.
`install_translator()` logs at INFO on success and WARNING on each failure mode;
read the QGIS log panel.

**Translations regressed after a release.**
Likely an uncommitted or stale `.qm`. `make i18n-check` warns about both; it is
advisory and easy to scroll past in `make package` output.

**Many entries went `vanished` after `make i18n-update`.**
Someone changed English source text, or renamed a class (which moves every one of
its strings to a new context — the old context's entries all become obsolete at
once). Qt Linguist can help re-pair them; check `git diff` on the `.ts` before
committing.
